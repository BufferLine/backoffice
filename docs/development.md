# Development

Local development environment setup.

## Prerequisites

- Python 3.12+
- Docker and Docker Compose
- Node.js 20+ (for frontend)
- Git

---

## Setup

```bash
# Clone
git clone https://github.com/YOUR_ORG/backoffice.git
cd backoffice

# Copy environment file
cp .env.example .env

# Full reset: starts Docker (PostgreSQL + MinIO), creates DB, runs migrations, starts server
scripts/reset.sh
```

After `reset.sh`, the following are running:
- Backend API: `http://localhost:8000`
- API docs (Swagger): `http://localhost:8000/docs`
- MinIO console: `http://localhost:9001`

### Install CLI (development mode)

```bash
cd cli
pip install -e "."
cd ..
```

### Initialize the system

```bash
acct init --api-url http://localhost:8000
# Open the printed URL to create your admin account
acct login --email admin@example.com --api-url http://localhost:8000
```

## Scripts

### `scripts/reset.sh` — Full reset

Wipes everything and starts fresh:

1. Stops all services and removes Docker volumes
2. Starts PostgreSQL and MinIO containers
3. Drops and recreates the database
4. Runs all Alembic migrations
5. Starts the backend server with `--reload`

Use this when you want a clean slate (e.g. after pulling schema changes).

```bash
scripts/reset.sh
```

### `scripts/dev.sh` — Start dev server

Starts services without wiping data. Runs any pending migrations and starts the backend with `--reload`.

```bash
scripts/dev.sh
```

### `scripts/stop.sh` — Stop everything

```bash
scripts/stop.sh
```

### `scripts/test.sh` — Run unit and integration tests

```bash
scripts/test.sh

# Pass pytest arguments
scripts/test.sh -k test_invoice
scripts/test.sh -x --tb=long
```

163 tests covering models, services, state machines, and API endpoints.

### `scripts/e2e.sh` — API E2E tests

Runs 31 end-to-end checks against a running server. Requires the server to be up.

```bash
scripts/e2e.sh
```

### `scripts/e2e-cli.sh` — CLI E2E tests

Runs 36 end-to-end checks through the `acct` CLI. Covers the full business workflow from init through month-end export. Requires the server to be up.

```bash
scripts/e2e-cli.sh
```

### `scripts/seed-demo.sh` — Seed demo data

Populates the database with sample clients, invoices, employees, payroll runs, and expenses.

```bash
scripts/seed-demo.sh
```

## Environment Variables

Copy `.env.example` to `.env`. Key variables:

| Variable | Description | Default (dev) |
|----------|-------------|---------------|
| `DB_PASSWORD` | PostgreSQL password | `devpassword123` |
| `MINIO_ACCESS_KEY` | MinIO access key | `minioadmin` |
| `MINIO_SECRET_KEY` | MinIO secret key | `miniodevpassword123` |
| `JWT_SECRET` | JWT signing key (32+ chars) | dev value |
| `DATABASE_URL` | Full DB connection string | set by scripts |
| `S3_ENDPOINT` | Object storage endpoint | `http://localhost:9000` |
| `S3_ACCESS_KEY` | S3 access key | same as MINIO_ACCESS_KEY |
| `S3_SECRET_KEY` | S3 secret key | same as MINIO_SECRET_KEY |
| `S3_BUCKET` | S3 bucket name | `backoffice` |
| `S3_REGION` | S3 region | `us-east-1` |
| `API_BASE_URL` | External URL (used in setup links) | `http://localhost:8000` |
| `CORS_ORIGINS` | Comma-separated allowed origins | `http://localhost:3000` |

## Adding New Features

### 1. Database changes

Create a new Alembic migration:

```bash
cd backend
source .venv/bin/activate
alembic revision --autogenerate -m "describe your change"
alembic upgrade head
```

Always review autogenerated migrations before applying. Check for missing `nullable=False` constraints or incorrect defaults.

### 2. Models

Add or modify a file in `backend/app/models/`. Import the model in `backend/app/models/__init__.py` so Alembic detects it.

### 3. Schemas

Add Pydantic schemas in `backend/app/schemas/`. Use separate `Create`, `Update`, and `Response` classes.

### 4. Services

Add business logic in `backend/app/services/`. Services receive a database session and return model instances or raise `HTTPException`.

### 5. API endpoints

Add a router in `backend/app/api/`. Register it in `backend/app/main.py` with the appropriate prefix and tags.

### 6. CLI commands

Add a command file in `cli/acct/commands/`. Register the `app` typer in `cli/acct/main.py`.

### 7. State machines

State transitions live in `backend/app/state_machines/`. Each machine maps action names to `{from_state: to_state}` dicts.

## Database Migrations

```bash
cd backend
source .venv/bin/activate

# Generate migration from model changes
DATABASE_URL=postgresql+asyncpg://backoffice:devpassword123@localhost:5432/backoffice \
  alembic revision --autogenerate -m "add my_table"

# Apply pending migrations
DATABASE_URL=postgresql+asyncpg://backoffice:devpassword123@localhost:5432/backoffice \
  alembic upgrade head

# Downgrade one step
DATABASE_URL=... alembic downgrade -1

# Show migration history
DATABASE_URL=... alembic history
```

## Code Conventions

- **Money fields**: `NUMERIC(19,6)` in DB, `Decimal` in Python — never `float`
- **Timestamps**: `TIMESTAMP(timezone=True)` (PostgreSQL TIMESTAMPTZ)
- **Primary keys**: UUIDs on all tables
- **State transitions**: return HTTP 409 on invalid state transitions
- **Audit log**: call `audit_log()` on all state-changing operations
- **Change log**: write field-level diffs on all PATCH operations
- **Idempotency**: `issue`, `finalize`, `export` commands are safe to retry
